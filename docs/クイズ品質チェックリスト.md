# クイズ品質チェックリスト（Ver. 2）

このドキュメントは、全ジャンル・全レベルのクイズ作成/監査時に必ず遵守するチェック観点をまとめたもの。
新規作成時だけでなく、既存データの監査（リビジョン）時にも本リストを基準にする。

## A. 問題文・ヒントの品質

- **A1. 自己リーク禁止**
  答えの単語（またはその明白な一部）が `question` / `hint1` 中に登場してはいけない。

- **A2. 事実正確性**
  問題文・ヒントに書かれている内容は生物学・歴史・地理などの最新の正確な情報に基づくこと。嘘や誇張、古い通説をそのまま書かない。

- **A3. レベル相応の語彙・知識**
  そのレベルの対象年齢/知識レベルにふさわしい語彙・内容にする（幼稚園児に専門用語を出さない、逆に社会人〜専門家レベルなのに簡単すぎる内容にしない）。

- **A4. 漢字＋ルビ**
  `question` / `hint1` は、そのレベルで習う範囲を超える漢字も含めて構わないが、必ず `{漢字|よみ}` 形式のルビを付与する。

- **A5. 問題文だけからの安易な推測を禁止（NEW）**
  問題文 (`question`) に含まれるキーワードの組み合わせだけで、ジャンル知識がなくても答えが一意に絞れてしまう作り方は禁止。特に **レベル6以上** は、問題文だけを読んで即答できるような強い特徴描写（学名的な分類キーワード、地域名＋生物名の直接的な組み合わせ、固有名詞そのものに近いキーワードなど）を並べすぎないこと。
  - ヒント (`hint1`) の方には、ある程度キーワード性の高い情報を残してよい（ヒントを読めば分かる、という難易度設計は許容）。
  - 目安: 問題文だけを読んで選択肢を見ずに答えられてしまうなら、その問題文はNG。選択肢と付き合わせて初めて絞り込めるくらいの情報量にする。

## B. 選択肢（ダミー）の品質

- **B1. 総称（generic term）を答え・選択肢にしない（NEW・強化）**
  正解が具体的な種・個体・固有の対象を指す問題では、**ダミー選択肢も正解と同じ具体性レベルの、実在する具体的名称**でなければならない。
  - NG例: 正解が「タイワンオオクワガタ」のような具体的な種名なのに、ダミーの1つが「クワガタ」（上位分類群・総称）になっている。総称は明らかに他の選択肢と毛色が違うため、消去法で正解が絞れてしまう。
  - NG例: 正解が「ジャイアントウェタ」（具体的な種）なのに、ダミーに「コオロギ」（総称）が含まれている。
  - **正解自体が総称になっている問題も禁止**。「わたしはだれ？」の答えが具体的な種名ではなく「コオロギ」のような広いカテゴリ名になっている問題は、具体的な種名の問題に差し替える。
  - 該当した場合は、同じ具体性レベルの実在する別の種・製品・人物名に差し替えること。

- **B2. ダミーは同じ文脈（属性）を共有すること（NEW・強化）**
  ダミー選択肢は、答えと**同じカテゴリ内の文脈属性**を共有し、その属性だけで消去法にならないようにする。
  - 歴史上の人物: 答えが日本人なら、ダミーも（少なくとも大半は）日本人にする。答えの時代に近い人物をダミーに含める。国籍や時代が違いすぎる人物を混ぜて、国籍/時代だけで正解以外を判別できてしまう構成はNG。
  - 生物: 答えと生息域・分類群がかけ離れすぎているダミーは避ける（完全に同じである必要はないが、属性だけで消去できてしまうほど極端な差は避ける）。
  - 道具・その他ジャンルも同様に、用途・分野・時代などの文脈が近いダミーを選ぶ。

- **B3. 近似重複ダミー禁止**
  ダミー同士、またはダミーと答えが、実質的に同じもの（別名・表記ゆれ・ほぼ同義語）にならないこと。

- **B4. ダミーの事実正確性**
  ダミーとして挙げる名称は実在し、スペル・表記が正しいこと。

- **B5. 漢字＋ルビ**
  `answer` / `dummy1〜3` も、漢字を含む場合は必ずルビを付与する（カタカナ・ひらがなのみのフィールドはルビ不要）。

## C. 全体整合性

- レベル内で `answer` の重複がないこと。
- 1問の中で4つの選択肢（正解＋ダミー3つ）が重複しないこと。

## 監査時の運用ルール

1. 新規ジャンル・新規レベルの問題を作成する際は、作成前に本チェックリストを踏まえて設計する。
2. 既存データを修正・追加する際も、触れた範囲は本チェックリスト（特にA5・B1・B2の新基準）で再チェックする。
3. 大規模監査（全ジャンル総ざらい）を行う場合は、対象ジャンル・レベルの優先順位をなつみに確認してから着手する。
4. 監査を始める前に必ず `docs/audit-status.json` を確認し、対象ジャンル×チェック項目が現行の `checklistVersion` で既に `status: "swept"` になっていれば、その組み合わせは再チェックしない（二重作業を避けるため）。監査・修正が完了したら、その場で `docs/audit-status.json` の該当エントリも更新すること（status・version・date・notesを記入）。
